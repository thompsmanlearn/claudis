## Task: Fix evaluator output_quality bug

### Problem diagnosed
agent_evaluator_4pillars (workflow kQ5OALBwexLQS7in) consistently scored `output_quality=3/5` with "no experimental outputs" even though experimental_outputs contained entries for evaluated agents. The BECOMING.md handoff from 2026-03-31 correctly identified the bug but hadn't diagnosed root cause.

### Root cause: 3 compounding issues found

**Issue 1 — fetch not defined in Code node sandbox (primary)**
The `Get Recent Outputs` Code node used `fetch()` to query Supabase. n8n Code node v2 runs in a vm2 sandbox with no access to `fetch`, `$helpers`, or `require`. The `try/catch` silently swallowed the `ReferenceError` and returned `outputs = []` every time. The evaluator had NEVER successfully fetched experimental_outputs in its entire existence.

**Issue 2 — output type pollution (secondary)**
Once Issue 1 was fixed, the top-5 most recent outputs for `agent_health_monitor` were all `4pillars_evaluation` records written by the evaluator itself. These pushed the actual `health_scan` operational outputs outside the `limit=5` window. The evaluator was eating its own output and drowning out the signal it was supposed to measure.

**Issue 3 — token truncation (tertiary)**
After fixing Issues 1 and 2, Haiku's response was cut off at `max_tokens=800` mid-JSON. With actual content in the prompt, the evaluation required more tokens to complete. Resulted in `parse_failed` fallback scores.

### Solution built

1. **Stats server `/get_outputs` endpoint** added to `stats_server.py`:
   - `GET /get_outputs?agent_name=X&limit=5&exclude_type=4pillars_evaluation`
   - Queries Supabase, returns `{outputs: [...], count: N}` — single JSON object, no array-unwrapping issue
   - `exclude_type` param filters out evaluator's own artifacts from output quality evidence

2. **Evaluator workflow restructured** (kQ5OALBwexLQS7in):
   - Added `Fetch Agent Outputs` HTTP Request node (calls `http://host.docker.internal:9100/get_outputs`)
   - Replaced `Get Recent Outputs` Code node with simple Code node reading `$('Fetch Agent Outputs').first().json.outputs`
   - Updated `max_tokens: 800 → 1500` in Build Eval Prompt node

3. **Stats server restarted** — confirmed new endpoint live

## Decisions

- **Stats server as Supabase proxy for n8n**: The correct pattern for any n8n node that needs to read from Supabase in a Code node context — add a stats server endpoint. Code nodes cannot use fetch. HTTP Request nodes return arrays that unwrap. Stats server endpoint returns a wrapped single object. This pattern is now proven across two use cases (memory_query, get_outputs).
- **exclude_type as standard parameter**: Any tool evaluating an agent's outputs should exclude meta-evaluation artifacts. The evaluator writing its own results into the evaluated agent's namespace is a design smell; the exclude_type filter is the pragmatic fix without restructuring the data model.
- **max_tokens calibration**: 800 tokens was adequate when evaluators found no outputs (short prompt). With actual experimental data, prompts grow. 1500 is the new baseline.
