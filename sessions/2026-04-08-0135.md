### 1. github_issue_tracker idempotency investigation — COMPLETE

**Question:** Session notes from 2026-04-07 flagged "github_issue_tracker had duplicate runs before idempotency kicked in — investigate whether the guard is reliable."

**Method:** Read full workflow JSON (F2lRufWUOXAGv5GB), inspected the idempotency mechanism, reviewed all 10 execution records, cross-referenced with audit_log entries.

**How the guard works:**
1. `Check Today's Scan` — GET experimental_outputs where `experiment_id = scan_{today}` and `output_type = issue_scan`
2. `Parse Idempotency` — sets `skip=true` if row exists
3. `Already Ran Today?` — routes to `Log Skip Audit` (skip=true) or `Fetch GitHub Issues` (skip=false)
4. On success, writes `experiment_id=scan_{today}` to experimental_outputs — which future triggers will find

**Execution history analysis:**
- April 1: 2 errors (21:57, 21:59) + 1 success (22:08) — initial build testing
- April 2: 6 executions (01:35, 01:39, 09:29, 09:30, 09:34, 09:37) — 5 passed guard, 1 correctly skipped (09:37)
- April 5: **1 execution** — clean, guard worked correctly

**Key finding:** April 2 duplicates were all during active build/testing (a Claudis session was iterating on the workflow). The April 5 run — the only real Sentinel production trigger — produced exactly 1 execution. The guard is reliable in production.

**Theoretical race condition:** If two Sentinel triggers fire within the ~2s check→write window, both could pass the guard before either writes the output row. This is real but extremely unlikely in normal Sentinel operation (Sentinel fires each agent once per window, not in bursts).
