### 1. memory_add MCP metadata bug — root cause confirmed and fixed

**Handoff item from 3 consecutive sessions.** The April 7 session documented the bug; April 8, 9, and 10 all carried it forward as "still live." This session traced it to root.

**Root cause:** Claude Code serializes JSON object MCP parameters as strings when calling MCP tools. When I call `mcp__aadp__memory_add` with `metadata: {"category": "ops"}`, Claude Code sends `metadata: '{"category": "ops"}'` (a string). ChromaDB rejects it with "not of type object."

**Fix applied:** `mcp-server/server.py`, `_memory_add` handler — added defensive coercion:
```python
if isinstance(metadata, str):
    try:
        metadata = json.loads(metadata)
    except (json.JSONDecodeError, TypeError):
        metadata = {}
```

**Verification:** Syntax OK. Coercion logic tested (str→dict confirmed). ChromaDB REST write with dict metadata confirmed (201). Fix takes effect on next MCP server restart. Not in git (mcp-server/ has no repo), but persists on disk.

**Note:** The input schema correctly declares `"metadata": {"type": "object"}`. The bug is in Claude Code's MCP client, not the schema. The fix makes the server robust regardless of whether the client honors the schema.
